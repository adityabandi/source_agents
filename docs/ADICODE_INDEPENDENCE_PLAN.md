# Adicode Independence & Rebrand Plan

**Goal:** Make the project fully independent of Claude/Anthropic and name it **adicode** everywhere.

This is a fork of Claude Code that was *partially* rebranded (the existing
`/tmp/rebrand.sh` only swapped `claude`→`adicode` in `src/`, leaving `anthropic`
untouched). The core still imports `@anthropic-ai/sdk`, calls `api.anthropic.com`,
and uses `anthropic-*` protocol headers. Per the decisions made:

1. **Backend:** Fully replace `@anthropic-ai/sdk` with a custom Adicode client + our own API contract.
2. **Contract:** New contract — we define our own request/response types; the client *translates* to whatever `adicode.ai` speaks.
3. **Rename scope:** Everything — identifiers, env vars, package name, binary, extension/bundle IDs.

---

## 0. Critical dependency / blocker

We do **not** yet have the `adicode.ai` wire contract. The plan therefore defines a
**canonical Adicode contract** (our types) and isolates the backend-specific mapping
in one `BackendAdapter`. Until the real `adicode.ai` spec is provided, the default
adapter targets the current Anthropic-compatible endpoint so the app keeps working;
it is swapped when the real contract lands. **Nothing else in the codebase talks to
the backend directly** — only the adapter does.

---

## 1. Architecture: the Adicode client

New module `src/services/api/adicodeClient.ts` (plus `adicodeTypes.ts`):

- `class Adicode` — mirrors the *subset* the codebase actually uses:
  - `adicode.beta.messages.create({...})` (non-streaming)
  - `adicode.beta.messages.stream(...)` / `create` returning an async-iterable of `AdicodeStreamEvent`
  - `adicode.beta.messages.countTokens({...})`
  - `asResponse()` access for raw headers (rate-limit parsing)
- `adicodeTypes.ts` — our own types replacing `Anthropic.*`:
  - `AdicodeMessageParam`, `AdicodeContentBlock`, `AdicodeToolUnion`, `AdicodeToolUseBlock`, `AdicodeToolResultBlock`, `AdicodeUsage`, `AdicodeStreamEvent`, `AdicodeBeta`, `AdicodeStopReason`, etc.
- Error classes `AdicodeError`, `AdicodeUserAbortError` replacing `APIError` / `APIUserAbortError`.
- **Translator layer:** builds our canonical request → calls `BackendAdapter.send()`.
  The adapter owns: base URL, auth header, `x-adicode-*` → wire header mapping,
  response → our canonical shape. Provider variants (bedrock/vertex/foundry) become
  adapter implementations behind the same `Adicode` interface (no `@anthropic-ai/*`
  imports in app code).

**Trade-off to confirm:** Replacing bedrock/vertex/foundry SDKs entirely is a large
lift. Recommended: implement `firstParty` (adicode.ai) with our own HTTP; implement
bedrock/vertex/foundry as thin adapters over our HTTP layer (drop `@anthropic-ai/*`
SDKs). Flagged for sign-off.

## 3. Risks & mitigations
- **Huge surface (300+ refs).** Mitigation: word-boundary sed scripts per phase + mandatory manual review of each `anthropic` hit (protocol vs. branding).
- **Unknown adicode.ai wire contract.** Mitigation: canonical contract + adapter seam; default adapter targets current endpoint until spec arrives (Phase 0 blocker noted above).
- **Type churn.** Replacing `Anthropic.*` types is mechanical but touches many files; do Phase B behind the new types and keep `typecheck` green per phase.
- **Bedrock/Vertex/Foundry SDKs.** Flagged trade-off in Phase 1 — confirm whether to fully drop or keep as adapters.

## 4. Rollout order
Phase A (foundation) → B (SDK swap) → C/D/E/F/G/H (renames) → I (verify). Each phase keeps `typecheck` green. Implementation belongs in **Code** mode; this document is the approved blueprint.
